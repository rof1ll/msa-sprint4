Ниже приведён реестр шагов для процесса обработки платежа в OrchestrPay. 

Предположения:
- списание денег у клиента рассматривается как шаг, который можно компенсировать автоматическим возвратом;
- перевод контрагенту считается необратимым в рамках текущей саги, в случае если после него потребуется возврат, это уже отдельный бизнес-процесс;
- изменения статусов и отправка уведомлений считаются идемпотентными, поэтому относятся к повторяемым транзакциям.

| Транзакция                   | Описание                                                                                               | Тип            | Компенсация               |
| ---------------------------- | ------------------------------------------------------------------------------------------------------ | -------------- | ------------------------- |
| `CREATE_PAYMENT`             | Создать запись платежа и зафиксировать старт Saga со статусом `NEW` или `PENDING`                      | компенсируемая | `CANCEL_PAYMENT`          |
| `DEBIT_CUSTOMER_ACCOUNT`     | Списать или зарезервировать деньги у клиента перед проверками                                          | компенсируемая | `REFUND_CUSTOMER_ACCOUNT` |
| `RUN_FRAUD_CHECK`            | Запустить antifraud/compliance-проверки и получить одно из решений: approve / reject / manual review   | повторяемая    | `—`                       |
| `START_MANUAL_REVIEW`        | Перевести платёж в состояние ожидания ручной проверки и создать задачу оператору                       | повторяемая    | `—`                       |
| `APPLY_CUTOFF_APPROVAL`      | При отсутствии ответа от проверяющих сервисов по cut-off-time автоматически считать платёж разрешённым | повторяемая    | `—`                       |
| `TRANSFER_TO_MERCHANT`       | Перевести деньги контрагенту после положительного решения проверок                                     | необратимая    | `—`                       |
| `MARK_PAYMENT_SUCCEEDED`     | Зафиксировать финальный успешный статус платежа во внутренней системе                                  | повторяемая    | `—`                       |
| `REFUND_CUSTOMER_ACCOUNT`    | Вернуть клиенту деньги, если платёж отклонён или произошёл сбой после списания                         | необратимая    | `—`                       |
| `MARK_PAYMENT_REJECTED`      | Зафиксировать финальный статус отклонённого/заблокированного платежа                                   | повторяемая    | `—`                       |
| `CANCEL_PAYMENT`             | Отменить платёж до списания средств или компенсировать шаг создания платежа                            | повторяемая    | `—`                       |
| `SEND_CUSTOMER_NOTIFICATION` | Отправить клиенту уведомление об успешном платеже, отклонении или возврате                             | повторяемая    | `—`                       |
| `SEND_SECURITY_ALERT`        | Отправить уведомление в систему безопасности о подозрительной операции                                 | повторяемая    | `—`                       |

## Логика компенсации

1. Если сбой произошёл после `CREATE_PAYMENT`, но до списания средств, достаточно выполнить `CANCEL_PAYMENT`.
2. Если сбой произошёл после `DEBIT_CUSTOMER_ACCOUNT`, но до `TRANSFER_TO_MERCHANT`, необходимо выполнить `REFUND_CUSTOMER_ACCOUNT`, затем `MARK_PAYMENT_REJECTED`.
3. Если `RUN_FRAUD_CHECK` вернул `reject`, запускается ветка `REFUND_CUSTOMER_ACCOUNT` -> `MARK_PAYMENT_REJECTED` -> `SEND_CUSTOMER_NOTIFICATION` -> `SEND_SECURITY_ALERT`.
4. Если `RUN_FRAUD_CHECK` вернул `manual review`, платёж переводится в ожидание через `START_MANUAL_REVIEW`; при таймауте применяется `APPLY_CUTOFF_APPROVAL`.
5. После `TRANSFER_TO_MERCHANT` текущая Saga считается доведённой до необратимого шага; дальнейшие возвраты post-factum должны оформляться отдельным процессом refund/reversal.
