
| Название функции                       | Назначение                                     | Что проверяется                                                                                                                                                | Действие при нарушении                                                                                                                    | Способ реализации                                                                                        |
| -------------------------------------- | ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| `SagaStateConsistency`                 | Консистентность состояний Saga                 | У каждой Saga есть допустимое состояние и корректный переход между шагами (`STARTED` -> `DEBITED` -> `APPROVED/REJECTED/MANUAL_REVIEW` -> финальное состояние) | Пометить Saga как `INCONSISTENT`, запустить восстановление состояния или компенсацию, отправить alert в support/ops                       | Scheduled job, который проверяет таблицу `saga_states` и журнал переходов по state machine               |
| `CrossServicePaymentStatusConsistency` | Консистентность данных между микросервисами    | Статус платежа в `Payment Service`, статус процесса в Camunda и записи об уведомлениях/возвратах не противоречат друг другу                                    | Создать incident, синхронизировать состояние через reconciliation job, при необходимости инициировать ручную проверку                     | Periodic reconciliation job, сравнивающий данные из `Payment DB`, Camunda/Operate API и журналов событий |
| `StuckSagaDetector`                    | Выявление «зависших» процессов                 | Процессы находятся в промежуточном состоянии (`PROCESSING`, `WAITING_MANUAL_REVIEW`, `REFUND_PENDING`) дольше допустимого SLA                                  | Создать incident, перевести процесс в recovery flow, при необходимости выполнить автоматическую компенсацию                               | Scheduled job + правило в мониторинге по возрасту записей в `saga_states` и `payments`                   |
| `PaymentTimeoutControl`                | Контроль времени выполнения платежей           | Платежи в статусе `PROCESSING` или `DEBITED` находятся в обработке дольше допустимого времени, например 30 минут                                               | Автоматический возврат средств, перевод платежа в `REJECTED/TIMED_OUT`, уведомление клиента                                               | Cron job, который ищет в БД платежи со статусом `PROCESSING`/`DEBITED` старше порога                     |
| `ManualReviewDeadlineControl`          | Контроль дедлайна ручной проверки              | User task ручной проверки не завершена за 20 минут                                                                                                             | Автоматически выполнить `APPLY_CUTOFF_APPROVAL` или перевести задачу в escalation flow, отправить alert                                   | Таймер Camunda + дополнительная проверка scheduled job по активным user task                             |
| `RefundAfterRejectGuarantee`           | Гарантия целостности компенсации               | Если antifraud вернул `reject` или произошёл сбой до pivot, то обязательно существует успешная операция `REFUND_CUSTOMER_ACCOUNT`                              | Повторно инициировать refund, заблокировать дальнейшие переходы Saga, создать критический alert                                           | Reconciliation job по связке `fraud_result`, `payment_status`, `refund_status`                           |
| `NoRefundAfterPivot`                   | Контроль pivot-точки Saga                      | После `TRANSFER_TO_MERCHANT` не запускается компенсация текущей Saga                                                                                           | Создать incident высокого приоритета, заблокировать автоматический refund в этой Saga, перевести кейс в отдельный refund/reversal process | Архитектурное правило + runtime-проверка по журналу шагов Saga                                           |
| `SingleFinalStateInvariant`            | Защита от двойного завершения процесса         | У одного платежа не может одновременно существовать более одного финального состояния (`SUCCEEDED`, `REJECTED`, `REFUNDED`)                                    | Пометить запись как corrupted, остановить дальнейшую обработку, отправить alert                                                           | SQL invariant check / scheduled data-quality job                                                         |
| `IdempotentDebitInvariant`             | Защита от двойного списания                    | Для одного `payment_id` или `idempotency_key` не более одного успешного `DEBIT_CUSTOMER_ACCOUNT`                                                               | Блокировать повторную операцию, создавать alert о нарушении идемпотентности                                                               | Проверка уникальных ключей в БД + integration/fitness job по журналу операций                            |
| `IdempotentRefundInvariant`            | Защита от двойного возврата                    | Для одного платежа не более одного успешного возврата по одной причине компенсации                                                                             | Блокировать повторный refund, создавать incident для финслужбы                                                                            | Уникальные ограничения + scheduled reconciliation по таблице `refunds`                                   |
| `CamundaVsDBProcessAlignment`          | Согласованность BPM-движка и доменной БД       | Для каждого активного процесса в Camunda есть соответствующий `payment` и `saga_state`, и наоборот                                                             | Повторная синхронизация, перезапуск оркестрации с сохранённого шага или создание incident                                                 | Scheduled job через Camunda Operate API/Zeebe API и запросы к PostgreSQL                                 |
| `NotificationDeliveryAfterFinalState`  | Контроль обязательных пост-действий            | После финального статуса (`SUCCEEDED` или `REJECTED`) клиенту отправлено уведомление; для `REJECTED` отправлен security alert                                  | Дозарегистрировать событие, повторить отправку уведомления, поднять warning/incident                                                      | Асинхронный outbox processor + reconciliation job по таблицам `notifications` и `alerts`                 |
| `ExternalCallRetryBudget`              | Контроль деградации внешних интеграций         | Количество retry для antifraud/compliance API не превышает допустимый лимит, а ошибки не приводят к бесконечным попыткам                                       | Остановить дальнейшие retry, перевести процесс в timeout/manual review, отправить alert                                                   | Метрики retry budget + policy в orchestrator/Camunda job worker                                          |
| `UnfinishedIncidentMonitor`            | Контроль необработанных технических инцидентов | В Camunda/Operate нет инцидентов старше допустимого порога                                                                                                     | Эскалировать в on-call, создать тикет, при необходимости инициировать recovery playbook                                                   | Alert rule в monitoring + polling Operate API по открытым incidents                                      |
| `OrphanPaymentDetector`                | Поиск «осиротевших» платежей                   | Есть запись о списании или возврате, но нет соответствующей Saga/процесса Camunda                                                                              | Заблокировать дальнейшую обработку, создать incident и запустить reconciliation                                                           | Scheduled cross-check между финансовыми таблицами и `saga_states`                                        |
| `BusinessDeadlineSLA`                  | Контроль соблюдения бизнес-SLA                 | 90% успешных платежей укладываются в целевое время обработки, например до 5 секунд                                                                             | Отправить alert о деградации SLA, инициировать анализ bottleneck                                                                          | Метрики Prometheus/Grafana + alerting rule по percentiles latency                                        |

## Примечания

1. Fitness-функции должны выполняться автоматически и регулярно, а не только во время ручной проверки.
2. Для критичных нарушений приоритетом считается финансовая консистентность, а не скорость завершения процесса.
3. Часть функций лучше реализуется как runtime-контроль в BPM-движке, часть — как reconciliation job поверх БД и журналов событий.
4. Минимальный набор для production: `SagaStateConsistency`, `StuckSagaDetector`, `RefundAfterRejectGuarantee`, `NoRefundAfterPivot`, `CamundaVsDBProcessAlignment`, `PaymentTimeoutControl`.
